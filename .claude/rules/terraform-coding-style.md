## Coding guidelines for Terraform

These are the coding guidelines for Terraform. The goal is to maintain consistency for all usage of Terraform throughout the organization.

### Conventions for files and directories

1. Use lowercase letters, dots and hyphens for file names.
2. Use lowercase letters and hyphens for directory names.
3. All modules must have: main.tf, outputs.tf, variables.tf
4. If main.tf grows to more than 200 lines, split it into smaller files.
   1. When splitting, attempt to group logically on the purpose of the resources and attempt to keep related resources together. Ex. if creating  a vm, all resources related to that vm should be in the same file. And could be named main.compute.tf. Use descriptive names.
   2. Do not split files by resource type. Ex. Do not create a file named main.azurerm_network_security_group.tf
   3. Do not create files named after the resource name. Ex. Do not create a file named main.my_nsg.tf

### Naming Conventions (for resources, variables, outputs)

1. Use only lowercase letters and underscores for resource names, variable names and outputs.
2. Use underscores to separate multiple words in names.
3. Resource names should have descriptive names and not be pluralized.
4. Use nouns for resource names and do not include the resource type in the name.
5. Wrap the resource type and name in double quotes in your resource definition.

### Resource Management

1. Prefer using `for_each` over `count` for resource creation. Except for single conditional resources, these should use `count`, and either be set to 0 or 1.
2. Group related resources together in the same file or module.
3. Use the `depends_on` attribute explicitly only when Terraform cannot automatically determine dependencies.
4. Implement proper deletion protection for critical resources.

### Code Documentation

1. Use # for single and multi-line comments.
2. README.md file in the root directory should provide an overview.
3. More detailed documentation can be added in the `docs/` directory.
4. Documentation in `*.md` files between the tags `<!-- BEGIN_TF_DOCS -->` and `<!-- END_TF_DOCS -->` are auto-generated by terraform-docs, do not edit them manually.
5. Document all non-obvious design decisions in code comments.
6. Add inline comments for complex expressions or workarounds.
7. Document any assumptions made in your Terraform code.

### Variables & Outputs

1. Include a type and description for every variable.
2. Always include detailed descriptions for variables and outputs.
3. Outputs should represent the key information needed by consumers.
4. Output primitive data types when possible (e.g., string, number, bool), not entire resource objects.
5. Use variable validation blocks to enforce constraints on input values.
6. Mark sensitive outputs appropriately with the `sensitive = true` attribute.
7. For complex variable types, include examples in the description.

### Code Organization & Formatting

1. Indent two spaces for each nesting level.
2. Use multiline heredoc syntax for long descriptions. These should follow this syntax:

    ```hcl
    variable "a_variable_name" {
      description = <<-DESCRIPTION
        Heredoc style should always start with '<<-DESCRIPTION'

        Over multiple lines if needed.

        Can contain code blocks, follow markdown syntax but do not add language specification.

        For example, this is a code block:

        ```
        #!/usr/env/bin bash

        # code block
        gh auth login
        ```

        Heredoc style should always end with 'DESCRIPTION' indented 2 spaces more than where the description declaration starts.
        DESCRIPTION
      type        = string
      nullable    = false
    }
    ```

3. When multiple arguments with single-line values appear on consecutive lines at the same nesting level, align their equals signs.
4. Use empty lines to separate logical groups of arguments within a block.
5. Avoid cluttering resource declarations with logic. Prefer to use locals or variables to simplify the resource declaration.
6. Prefer to order resources in the order they are created. With the goal that each `*.tf` file can be read by a human as a recipe/script from top to bottom. Data resources should be declared just before the resources that use them.
7. Avoid making `*.tf` files too large to be comprehensible by humans. Break it into smaller files if necessary grouped logically by functionality. Use a consistent naming convention like `main.nsgs.tf`, `main.vnet.tf`, etc. Applies to `main.tf`, `variables.tf` and `outputs.tf`.
8. Use format on save in your editor. This ensures `terraform fmt` is run on every save.

### Error Handling

Terraform provides a layered validation framework. Use these mechanisms — in order of when they execute — to make errors visible early and with clear messages. Prefer explicit validation over silent fallbacks.

1. **Input variable validation** — runs before plan, blocks the operation if the condition fails. Use for all inputs that have constraints.

    ```hcl
    variable "environment" {
      type        = string
      description = "Deployment environment."
      validation {
        condition     = contains(["dev", "staging", "prod"], var.environment)
        error_message = "environment must be one of: dev, staging, prod."
      }
    }
    ```

2. **Preconditions** — run during plan, before a resource or data source is created. Use to verify assumptions about data sources or combinations of input values that cannot be expressed as a single variable validation.

    ```hcl
    resource "azurerm_linux_virtual_machine" "main" {
      # ...
      lifecycle {
        precondition {
          condition     = var.os_disk_size_gb >= 30
          error_message = "os_disk_size_gb must be at least 30 GB."
        }
      }
    }
    ```

3. **Postconditions** — run after a resource is created or read. Use to verify guarantees that must hold after Terraform has acted, so downstream resources fail fast with a clear message rather than a cryptic provider error.

    ```hcl
    data "azurerm_key_vault_secret" "db_password" {
      # ...
      lifecycle {
        postcondition {
          condition     = self.value != ""
          error_message = "Key Vault secret '${self.name}' is empty."
        }
      }
    }
    ```

4. **Check blocks** — run after plan/apply and issue warnings without blocking the operation. Use for continuous health assertions, not for enforcing correctness.

    ```hcl
    check "storage_account_accessible" {
      assert {
        condition     = azurerm_storage_account.main.primary_blob_endpoint != ""
        error_message = "Storage account has no accessible blob endpoint."
      }
    }
    ```

### Anti-Patterns

#### `try()` — avoid except for data normalization

`try()` evaluates an expression and returns a fallback value instead of raising an error. HashiCorp explicitly warns against overusing it: *"Overuse of `try` to suppress errors will lead to a configuration that is hard to understand and maintain."*

Only use `try()` in a local value whose sole purpose is normalising external or dynamic input (e.g., decoded YAML/JSON with an unpredictable schema). Isolate it in one place so the rest of the module can use plain references.

```hcl
# OK — normalising decoded external data in a single local
locals {
  raw    = yamldecode(file("${path.module}/config.yaml"))
  config = {
    name   = tostring(try(local.raw.name, "default"))
    groups = try(local.raw.groups, [])
  }
}
```

Do **not** scatter `try()` through resource declarations or use it to paper over missing variables, incorrect types, or logic mistakes. These are bugs that should surface immediately.

```hcl
# Bad — hides the real problem; if var.settings is wrong the error disappears
subnet_id = try(var.settings.subnet_id, null)
```

Note that `try()` only catches *dynamic* runtime errors. It cannot suppress errors for provably invalid constructs such as references to undeclared variables — those still fail at parse time.

#### `can()` — only in variable validation blocks

`can()` returns a boolean indicating whether an expression succeeds. Its intended and only recommended use is in `validation` blocks to perform type checks on input variables.

```hcl
# OK — type-checking a timestamp in a validation block
variable "expires_at" {
  type        = string
  description = "RFC 3339 expiry timestamp."
  validation {
    condition     = can(formatdate("", var.expires_at))
    error_message = "expires_at must be a valid RFC 3339 timestamp."
  }
}
```

Do **not** use `can()` in resource declarations, locals, or module logic outside of validation blocks. Anywhere else, prefer explicit conditionals or preconditions.

#### Overly complex conditionals

Nested ternary expressions are hard to read, test, and refactor. Move complex branching into a local lookup map.

```hcl
# Bad
instance_type = var.env == "prod" ? (var.size == "large" ? "Standard_D4s_v3" : "Standard_D2s_v3") : "Standard_B2s"

# Good
locals {
  instance_type = {
    prod    = { large = "Standard_D4s_v3", small = "Standard_D2s_v3" }
    default = { large = "Standard_B2s",    small = "Standard_B2s" }
  }
}
vm_size = local.instance_type[contains(["prod"], var.env) ? var.env : "default"][var.size]
```

#### Unvalidated dynamic blocks

Dynamic blocks built from unvalidated input can silently produce incorrect infrastructure. Always define a typed `object` variable with a `validation` block before using the variable in a `dynamic` block.

```hcl
# Bad — no constraints on what values are allowed
dynamic "ip_rule" {
  for_each = var.ip_rules
  content {
    ip_address = ip_rule.value
  }
}

# Good — validate structure and values before they reach the resource
variable "ip_rules" {
  type        = list(string)
  description = "List of allowed IP addresses in CIDR notation."
  validation {
    condition     = alltrue([for ip in var.ip_rules : can(cidrnetmask(ip))])
    error_message = "All entries in ip_rules must be valid CIDR blocks."
  }
}
```
